动态指令路由：
给大模型的技能操作手册体系

这不是 Function Calling，也不是 RAG。这是一套完全不同的 Agent 扩展范式——把专业领域的操作指南以 Markdown 文件的形式组织起来，在运行时动态注入模型上下文，让模型「读完手册后自己去做事」。

01

两种范式的根本区别

在讨论这套体系之前，必须先厘清一个极容易混淆的概念。Agent 扩展能力的主流方式有两种，它们的工作方式截然不同：

维度 工具调用范式（Tool Calling） 指令路由范式（Skill Routing）本文主角
Skill 的本质 可执行的外部函数，有 JSON Schema 给模型读的操作手册，是 Markdown 文件
触发方式 模型输出结构化 tool_use 请求 模型匹配描述后，将文件内容加载进上下文
执行者 平台 / 服务器代码 模型自己，按照手册步骤行事
扩展的是 模型能访问的外部资源和能力 模型完成复杂任务的方式和专业深度
典型用途 查天气、调 API、操作数据库 生成报告、执行专业流程、多步骤任务编排

指令路由范式的核心洞见是：模型最大的瓶颈不是缺少外部工具，而是缺少处理特定领域任务的专业指导。一个会使用代码执行工具的模型，不一定知道怎样构建一份合格的数据分析报告；一个能搜索互联网的模型，不一定知道撰写技术文档的最佳实践。Skill Routing 解决的正是这个问题。

一句话定义

Skill = 一份领域专家写给模型看的「任务操作手册」，在运行时按需加载进模型上下文，引导模型用专业方式完成特定类型的任务。

02

三级加载机制

这套体系的架构核心是「渐进式上下文加载」。不是把所有 Skill 的完整内容都塞进每次对话，而是分三个层次按需加载，在上下文预算和能力覆盖之间取得最优平衡：

①

元数据层（Metadata）

只有 name 和 description 两个字段，始终存在于模型上下文。这是模型判断「该不该用某个 Skill」的唯一依据。所有 Skill 的元数据以列表形式注入，让模型知道自己有哪些可用技能。

始终加载 ~100 词
②

指令正文层（SKILL.md Body）

当模型判断当前任务匹配某个 Skill 时，将该 Skill 的完整 Markdown 指令文件加载进上下文。这是模型执行任务时的操作指南，包含步骤、格式要求、示例、注意事项等。

按需加载 < 500 行
③

捆绑资源层（Bundled Resources）

SKILL.md 中引用的附加资源：可执行脚本（scripts/）、参考文档（references/）、模板文件（assets/）。由模型在执行过程中按照指令主动调取，无需预先全部加载。容量不限。

执行中调取 不限容量

三级机制的精妙在于：无论平台注册了多少个 Skill，对每次对话的上下文消耗都是可控的。只有真正被激活的 Skill 才会完整加载，其余全部以轻量元数据存在。

设计取舍

SKILL.md 正文建议控制在 500 行以内。如果一个 Skill 过于复杂，应当将细节拆分到 references/ 目录的子文档中，并在 SKILL.md 里提供清晰的引用指引，而不是在单文件中无限扩张。

03

SKILL.md 文件规范

每个 Skill 是一个独立目录，以 SKILL.md 为核心，可按需附带资源文件。目录结构如下：

my-skill/
  SKILL.md         ← 必须
  scripts/         ← 可执行代码，避免重复造轮子
    build_report.py
    validate_schema.js
  references/      ← 按需读入的参考文档
    aws.md
    gcp.md
  assets/         ← 模板、字体、图标等输出素材
    report_template.docx

SKILL.md 文件结构

文件由两部分构成：YAML 前置元数据（frontmatter）和 Markdown 指令正文。

---
name: quarterly-report-generator
description: |
  生成季度业务分析报告，包含数据可视化、趋势分析和管理层摘要。
  当用户提到季报、Q1/Q2/Q3/Q4报告、业务回顾、数据汇总成报告、
  需要把数据整理成正式文档时，使用此 Skill。即使用户只是说"帮
  我整理一下这些数据"而上传了CSV文件，也应考虑激活此 Skill。
compatibility:
  tools: [code_execution, file_read]   # 可选，说明所需工具
---

# 此处是 Markdown 正文，即模型读到的操作指南

## 输入处理
接收用户上传的数据文件（CSV/Excel/JSON）。
若文件编码异常，使用 scripts/detect_encoding.py 预处理。

## 执行步骤
1. 读取并清洗数据，记录缺失字段和异常值
2. 调用 scripts/build_charts.py 生成图表
3. 按照下方模板结构撰写报告正文
4. 使用 assets/report_template.docx 输出最终文件

## 报告结构模板
始终遵循此结构：
- 执行摘要（3-5句话，面向管理层）
- 核心指标（表格，含环比变化）
- 趋势分析（结合图表解读）
- 下季度建议（具体可操作的3条）

## 参考文档
不同数据来源的字段映射见 references/ 目录：
- Salesforce 导出数据 → 读取 references/salesforce-schema.md
- Google Analytics 导出 → 读取 references/ga4-schema.md

04

指令正文的编写模式

SKILL.md 的正文不是写给工程师看的技术文档，它是写给模型看的操作手册。如何让模型读完之后能高质量地完成任务，是这里最关键的工程问题。

原则一：解释「为什么」比下命令更有效

模型不是执行脚本的解释器，它是一个有推理能力的智能体。当指令说明背后的原因，模型能更灵活地应对边缘情况，而不是机械地套用规则。

// ❌ 命令式，脆弱，遇到变体就失效
报告标题必须使用 H1 格式，不得使用其他格式。

// ✅ 解释原因，模型能举一反三
报告的标题层级应保持一致性（通常用 H1 作顶层），
因为最终文档会被转换为 PDF，层级混乱会导致目录生成错误。
writing-style

原则二：用示例传达格式意图

模式匹配是大模型的强项。与其用文字描述格式规则，不如直接给出输入 → 输出的示例：

## 提交信息格式示例

**示例 1：**
输入：新增了用 JWT 做用户认证的功能
输出：feat(auth): implement JWT-based user authentication

**示例 2：**
输入：修了登录页面在 Safari 上闪烁的问题
输出：fix(ui): resolve login page flicker on Safari

# 格式规律：type(scope): lowercase description
# 模型通过示例自行推断，无需逐条列出规则
examples-pattern

原则三：脚本化重复劳动

如果测试运行中发现模型在每次执行时都独立编写同一段辅助代码（比如每个 eval 都写了 create_chart.py），这是明确信号：应该把这段代码写入 scripts/ 目录，并在 SKILL.md 里告知模型直接调用，避免每次重新发明轮子。

原则四：大型参考文档的组织方式

当一个 Skill 需要覆盖多个子域（例如云部署支持 AWS / GCP / Azure），不要把所有内容堆在 SKILL.md 里，而是分文件组织，在主文件中明确指引何时读哪个：

## 部署目标选择
根据用户指定的云平台，读取对应的参考文档后再继续：
- AWS → 读取 references/aws.md（含 IAM 权限配置和 VPC 网络设置）
- GCP → 读取 references/gcp.md（含 Service Account 和 Cloud Run 配置）
- Azure → 读取 references/azure.md（含 AD 集成和 ARM 模板）

若用户未指定，询问后再继续。不要同时加载多份文档。
domain-routing
渐进式披露原则

对于超过 300 行的参考文档，在文档头部提供一个简短目录，让模型能快速定位需要的章节，而不必通读全文。这与写给人类读者的长文档最佳实践一致。

05

触发机制深度解析

整套体系中最微妙、也最影响实际效果的部分，就是 description 字段。它是模型决定「要不要激活这个 Skill」的唯一依据，其质量直接决定系统的可用性。

触发的底层工作原理

系统运行时，所有可用 Skill 的元数据（name + description）会以列表形式注入 System Prompt，形如：

// System Prompt 中的 available_skills 部分（示意）

你有以下可用技能，在任务需要时请主动调用：

<available_skills>
  <skill name="quarterly-report-generator">
    生成季度业务分析报告，包含数据可视化、趋势分析和管理层摘要。
    当用户提到季报、Q1/Q2/Q3/Q4报告、业务回顾、数据汇总成报告时使用...
  </skill>
  <skill name="code-review-assistant">
    执行全面的代码审查，涵盖安全漏洞、性能问题、代码规范...
  </skill>
  // ... 更多 Skill 元数据
</available_skills>

当前用户消息："帮我把上个季度的销售数据整理成报告"

// 模型读取元数据，判断 quarterly-report-generator 匹配
// 平台加载对应 SKILL.md 完整内容，注入上下文
// 模型按照 SKILL.md 指令执行任务
trigger-mechanism

写好 description 的三个维度

有效 description 的结构模板

  • 是什么（What）：Skill 的能力摘要，一句话说清楚能做什么、产出什么
  • 何时用（When）：列举触发场景，用「当用户...时」的句式，覆盖不同说法（正式 / 口语 / 隐含需求）
  • 何时不用（When NOT）：负例约束，避免与相似 Skill 产生混淆或误触发
// ❌ 过于简短，触发率低，且无法区分竞争 Skill
description: "生成报告"

// ✅ 完整的三维描述
description: |
  将原始数据（CSV/Excel/JSON）转化为结构化的季度业务分析报告，
  包含图表、趋势分析和管理层摘要，最终以 Word/PDF 格式输出。

  触发场景：用户提到季报、月报、Q1-Q4 报告、业务复盘、数据整理
  成文档、"帮我把这些数字做成报告"，或上传了数据文件并希望输出
  正式报告的任何情形。即使用户没有明确说"报告"，只要意图是把
  数据转化为可分享的正式文档，都应激活此 Skill。

  不适用：数据探索分析（用 data-explorer Skill）、实时数据大屏
  （用 dashboard Skill）、单纯的数据清洗（无需报告输出时）。
description-best-practice

「欠触发」问题与反制策略

在实践中，模型普遍存在「欠触发」倾向：它倾向于自己直接回答，而不去查询是否有更合适的 Skill。对抗这一倾向的策略是在 description 中适当添加「主动推荐」的语气：

// 在 description 末尾添加主动推荐语

  ...即使用户只是模糊地说"帮我整理一下"而没有明确要求格式，
  只要上下文涉及数据汇报，也应主动考虑激活此 Skill，而不是
  仅仅给出文字分析。

trigger-push
重要：description 和 SKILL.md 正文的分工

所有「何时使用」的信息只应放在 description（frontmatter）中，不要放在 SKILL.md 正文里。正文是任务执行手册，不是触发判断依据。模型在读取 SKILL.md 之前，只能看到 description——正文里写的触发条件永远不会被读到。

在自己的 Agent 平台上实现这套体系，核心工作是构建一个「Skill 路由层」，负责管理元数据注入、响应 Skill 加载请求、维护上下文预算。以下是完整的实现蓝图。

平台整体分层

Skill Registry 核心接口

class SkillRegistry {
  /**
   * 获取所有元数据，用于注入 System Prompt
   * 这个列表始终在模型上下文中
   */
  async getAllMetadata(userId): Promise<SkillMeta[]> {
    return db.query(`
      SELECT name, description 
      FROM skills 
      WHERE (owner_id = $1 OR visibility = 'public')
        AND is_active = true
      ORDER BY usage_count DESC
    `, [userId]);
  }

  /**
   * 加载完整 SKILL.md 内容
   * 在模型决定激活某 Skill 时调用
   */
  async loadSkillContent(skillName: string): Promise<SkillContent> {
    const skill = await db.skills.findByName(skillName);
    const markdown = await storage.readFile(`${skill.path}/SKILL.md`);
    const resources = await storage.listDir(`${skill.path}/`);
    return { markdown, resources, skillPath: skill.path };
  }
}

System Prompt 构建器

async function buildSystemPrompt(userId: string, basePrompt: string) {
  // 1. 获取用户可用的所有 Skill 元数据
  const skills = await registry.getAllMetadata(userId);

  // 2. 构建 available_skills 块注入 System Prompt
  const skillsBlock = skills.map(s => `
  <skill name="${s.name}">
    ${s.description}
  </skill>`).join('\n');

  return `${basePrompt}

<available_skills>
当你判断当前任务适合使用某个技能时，请先声明你要使用的技能名称，
格式：<use_skill name="skill-name" />
系统会自动加载该技能的完整操作手册到你的上下文中。
${skillsBlock}
</available_skills>`;
}

// 3. 检测模型响应中的 use_skill 声明
function detectSkillActivation(response: string): string | null {
  const match = response.match(/<use_skill name="([^"]+)"/);
  return match ? match[1] : null;
}

// 4. 激活 Skill：将 SKILL.md 注入并继续对话
async function activateSkill(skillName: string, messages: Message[]) {
  const { markdown, skillPath } = await registry.loadSkillContent(skillName);

  // 把 SKILL.md 内容作为 tool_result 或 assistant 注入
  messages.push({
    role: 'user',
    content: `[Skill Loaded: ${skillName}]\n\n${markdown}\n\n---\n请按照上述操作手册继续完成任务。技能资源路径：${skillPath}`
  });

  return messages;
}
system-prompt-builder.ts

完整的对话处理循环

01

构建 System Prompt，注入所有元数据

将用户可用 Skill 的 name + description 列表注入 System Prompt，作为本次对话的初始上下文。

02

模型第一次响应：判断是否激活 Skill

模型读取元数据列表后，决定当前任务是否匹配某个 Skill。如果匹配，输出 <use_skill name="..." /> 声明。

03

平台检测声明，加载 SKILL.md

路由层解析模型响应，检测到 use_skill 声明后，从存储中读取完整的 SKILL.md 文件内容，注入对话上下文。

04

模型第二次响应：按手册执行任务

模型现在能看到完整的操作手册，按照其中的步骤、格式要求和引用资源完成任务。如需捆绑资源，会主动发出读取请求。

05

资源按需调取

模型在执行过程中若需要 references/ 中的参考文档或 scripts/ 中的脚本，平台响应读取请求，将对应文件内容追加到上下文。

07

支持用户上传 Skill 的平台设计

让用户能够自定义并上传 Skill，是这套体系真正爆发价值的关键。用户最了解自己领域的操作规范，他们创建的 Skill 往往比平台预置的更贴合实际需求。

Skill 上传后的验证流程

01

结构校验

检查是否存在 SKILL.md，YAML frontmatter 是否合法，必填字段 name 和 description 是否存在。

02

Description 质量评分

用一个轻量模型评估 description 的触发覆盖度——是否包含触发场景描述、是否有负例约束、长度是否合适（建议 50~300 词）。评分不足时给出改进建议。

03

安全审查

扫描正文内容，确认不包含恶意提示词注入、不试图覆盖 System Prompt 的安全策略、捆绑脚本需经代码安全扫描。

04

沙箱测试运行

使用用户提供的（或自动生成的）测试提示词，在沙箱环境中运行一次，确认 Skill 能被正常激活和执行，生成预览结果供用户确认。

05

注册入库

将 SKILL.md 存入文件存储，元数据写入数据库，同时对 description 文本做向量化，写入向量索引，用于规模化场景下的语义检索。

数据库 Schema

CREATE TABLE skills (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name         VARCHAR(100) NOT NULL UNIQUE,
  display_name VARCHAR(200),

  -- 元数据层（始终在内存中，用于注入 System Prompt）
  description  TEXT NOT NULL,
  description_embedding vector(1536),  -- 用于语义检索

  -- 文件层
  skill_path   TEXT NOT NULL,   -- 指向存储中的 SKILL.md 路径
  has_scripts  BOOLEAN DEFAULT false,
  has_refs     BOOLEAN DEFAULT false,
  has_assets   BOOLEAN DEFAULT false,

  -- 权限与版本
  owner_id     UUID REFERENCES users(id),
  visibility   VARCHAR(20) DEFAULT 'private',
  version      VARCHAR(20) DEFAULT '1.0.0',
  is_active    BOOLEAN DEFAULT true,

  -- 统计（用于排序和推荐）
  usage_count  INTEGER DEFAULT 0,
  avg_rating   FLOAT,
  created_at   TIMESTAMPTZ DEFAULT NOW()
);

-- 用户安装的 Skill（多对多，支持个性化配置）
CREATE TABLE skill_installations (
  user_id      UUID REFERENCES users(id),
  skill_id     UUID REFERENCES skills(id),
  is_enabled   BOOLEAN DEFAULT true,
  installed_at TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (user_id, skill_id)
);

08

Description 的系统性优化

Skill 上线之后，description 的触发准确率往往需要持续调优。这个过程可以系统化：构建一批「触发评估查询集」，用它来量化 description 的好坏，并指导优化方向。

构建触发评估集

自动优化循环

有了评估集，就可以构建自动优化循环：用模型本身来评估当前 description 在测试集上的表现，再用另一个模型来提出改进方案，迭代直到准确率收敛：

async function optimizeDescription(skillPath, evalSet, maxIterations = 5) {
  let currentDesc = parseSkillDescription(skillPath);
  let bestDesc = currentDesc;
  let bestScore = 0;

  // 将评估集拆分：60% 训练集，40% 留存测试集
  const { train, test } = splitEvalSet(evalSet, 0.6);

  for (let i = 0; i < maxIterations; i++) {
    // 1. 用训练集评分（每个查询跑 3 次取均值，减少随机性）
    const trainScore = await evaluateDescription(currentDesc, train, runs=3);
    const testScore  = await evaluateDescription(currentDesc, test, runs=3);

    // 2. 用测试集分数决定是否保存（防止过拟合训练集）
    if (testScore > bestScore) {
      bestScore = testScore;
      bestDesc = currentDesc;
    }

    // 3. 分析失败案例，让模型提出改进
    const failures = getFailedCases(currentDesc, train);
    currentDesc = await proposeImprovedDescription(currentDesc, failures);

    console.log(`Iter ${i+1}: train=${trainScore.toFixed(2)} test=${testScore.toFixed(2)}`);
  }

  return { best_description: bestDesc, best_score: bestScore };
}

最终建议

Description 优化是一个持续过程，而不是一次性工作。随着用户使用 Skill 的方式积累，你会发现新的触发模式和遗漏场景。建议定期（如每月一次）用真实的对话日志补充评估集，持续迭代 description 质量。
